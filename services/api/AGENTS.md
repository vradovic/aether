# Aether API Working Guide

## Read this first

Every agent or contributor working anywhere under `services/api/` must read this file before inspecting, editing, generating, or testing API code. This file applies to the entire API module. A more deeply nested `AGENTS.md`, if one is added later, may provide additional or overriding instructions for its subtree.

Work from `services/api/` unless a command says otherwise. Also respect any repository-level instructions above this directory.

## Project purpose

Aether is a cloud-native, real-time communication system. This directory contains its Go backend, currently built as a modular monolith: one deployable application composed of feature-focused internal modules.

The backend is expected to handle concerns including:

- Authentication, credentials, sessions, and authorization.
- Users and profiles.
- Real-time messaging over WebSockets.
- Conversations, membership, presence, and message persistence.
- Supporting HTTP APIs, configuration, observability, and database access.

Keep module boundaries clear enough that features can evolve independently, but do not introduce distributed-system or microservice abstractions before the application needs them.

## Current structure

```text
services/api/
|-- cmd/api/                 # Application entry point
|-- deployments/            # Local deployment configuration
|-- docs/
|   `-- openapi.yaml         # Canonical HTTP API contract
|-- internal/
|   |-- auth/                # Authentication HTTP and business logic
|   |-- config/              # Environment-backed configuration
|   |-- db/                  # sqlc-generated database package
|   |-- server/              # Dependency wiring, routes, server lifecycle
|   `-- users/               # User HTTP and business logic
|-- sql/
|   |-- migrations/          # Goose migrations
|   `-- queries/             # sqlc query definitions
|-- go.mod
`-- sqlc.yaml
```

Add a new feature as a cohesive package under `internal/<feature>` when practical. Keep `cmd/api` small and use `internal/server` as the composition root.

## Architectural conventions

Follow the existing handler-service-database flow:

```text
HTTP/WebSocket request
    -> feature handler
    -> feature service/use case
    -> narrow database or infrastructure interface
    -> sqlc/pgx implementation
```

### Handlers

- Own transport concerns: routing, body limits, JSON decoding/encoding, headers, status codes, and WebSocket protocol details.
- Define request and response types that describe the external API contract, for example `registerRequest` and `userResponse`.
- Convert transport types into service inputs explicitly.
- Translate service errors into stable HTTP or WebSocket responses.
- Do not contain business rules or expose raw database errors and generated database rows as API contracts.

### Services

- Own use-case behavior, normalization, business validation, authorization decisions, and orchestration.
- Name inputs after the use case, for example `registerInput`, rather than after a database query.
- Normalize before validating. Preserve meaningful data such as password whitespace.
- Accept `context.Context` as the first parameter and pass it through to I/O operations.
- Depend on small interfaces containing only the methods the feature consumes. Define those interfaces near the consuming service.
- Convert service inputs to `db.*Params` only at the persistence boundary.

Example:

```go
type registerInput struct {
	email     string
	password  string
	firstName string
	lastName  string
}

type querier interface {
	CreateUser(context.Context, db.CreateUserParams) error
}

func (s *service) register(ctx context.Context, input registerInput) error {
	input = input.normalize()
	if err := input.validate(); err != nil {
		return err
	}

	// Perform the use case and map to db.CreateUserParams here.
	return nil
}
```

### Database and migrations

- Treat `internal/db/*.go` files marked `Code generated by sqlc. DO NOT EDIT.` as generated artifacts.
- Change SQL in `sql/queries` or `sql/migrations`, then run `sqlc generate` from this directory.
- Keep schema invariants in migrations even when the service also validates them.
- Use a new timestamped Goose migration for schema changes; do not rewrite a migration that may already have been applied outside local development.
- Keep transactions at the service/use-case boundary when an operation requires multiple writes to succeed atomically.

### API documentation

- Treat `docs/openapi.yaml` as the canonical HTTP API contract and keep it aligned with the behavior implemented by handlers.
- Any change that adds, removes, or modifies an HTTP endpoint, request, response, status code, authentication requirement, or externally visible validation rule must update `docs/openapi.yaml` in the same change.
- Include realistic, non-secret examples for request bodies, parameters, successful responses, and important error responses.
- Keep operation IDs stable after clients begin depending on them.
- Document future WebSocket upgrade endpoints in OpenAPI. Add an AsyncAPI document when message types and bidirectional event flows are introduced.

### Errors, logging, and security

- Use stable sentinel or typed service errors when callers need to branch with `errors.Is` or `errors.As`.
- Translate pgx/PostgreSQL errors inside or immediately behind the service boundary; handlers should not need database-specific knowledge.
- Return safe client messages and log internal details with `log/slog` key-value fields.
- Never log plaintext passwords, tokens, authorization headers, private message content unnecessarily, or connection strings containing credentials.
- Hash credentials only in the service layer. Respect bcrypt's 72-byte password limit.
- Apply request size limits and explicit server timeouts. For WebSockets, define limits, deadlines, cancellation, backpressure, and single-writer ownership deliberately.

## Go style

- Prefer the standard library and small, explicit types over framework-style abstractions.
- Follow idiomatic Go naming: initialisms such as `ID`, `HTTP`, `URL`, and `DTO` stay capitalized. Prefer semantic names such as `registerRequest` over generic names such as `registerDTO`.
- Keep exported APIs minimal. Do not export a type or method only to make it testable.
- Return errors instead of panicking during normal request processing.
- Wrap errors with useful context using `%w` when the additional context helps and callers must retain the underlying error.
- Avoid package-level mutable state and hidden goroutine lifetimes.
- Keep functions focused, but do not extract helpers or generalized validation machinery until repetition or a real shared policy justifies it.
- Document exported identifiers when their behavior is not self-evident.
- Preserve unrelated work in the working tree and keep changes scoped to the task.

## Testing conventions

- Place tests beside the source as `*_test.go`.
- Use `package <feature>` for focused tests of intentionally unexported behavior.
- Use `package <feature>_test` when testing only the package's public contract. Do not create a separate `tests` directory without a concrete integration-test need.
- Prefer table-driven tests for validation and boundary cases.
- Use small fakes implementing the service's consumer-owned interfaces. Verify both outputs and important side effects, such as whether a database method was called.
- Use `net/http/httptest` for handler and routing behavior.
- Test normalization through the service as well as in isolation so ordering mistakes are caught.
- For authentication, test that plaintext credentials are never persisted and that produced hashes verify correctly.
- For future concurrent/WebSocket code, test cancellation and disconnect paths and run the race detector.

## Common tools and commands

Run these from `services/api/` and narrow them during iteration when useful.

```sh
# Discover code quickly
rg --files
rg "pattern" internal sql

# Format every Go file changed by the task
gofmt -w internal/auth/service.go internal/auth/service_test.go

# Run one package or one test while iterating
go test ./internal/auth
go test ./internal/auth -run TestServiceRegister

# Required broad verification for ordinary Go changes
go test ./...
go vet ./...

# Important for new concurrent or WebSocket behavior
go test -race ./...

# Update module metadata only when dependencies changed
go mod tidy
go mod verify

# Regenerate database code after SQL query/schema changes
sqlc generate

# Load the local database URL while tolerating CRLF line endings in .env
export DB_ADDRESS="$(sed -n 's/^DB_ADDRESS=//p' .env | tr -d '\r')"

# Create and validate timestamped Goose migrations
goose -dir sql/migrations create <migration_name> sql
goose -dir sql/migrations validate

# Inspect, apply, or roll back local database migrations
goose -dir sql/migrations postgres "$DB_ADDRESS" status
goose -dir sql/migrations postgres "$DB_ADDRESS" up
goose -dir sql/migrations postgres "$DB_ADDRESS" down

# Start the local PostgreSQL dependency
docker compose --env-file deployments/.env -f deployments/docker-compose.yml up -d

# Open the generated API reference at http://localhost:8081
docker compose --env-file deployments/.env -f deployments/docker-compose.yml up -d aether-swagger-ui
```

Use `go test ./...` and `go vet ./...` before handing off code unless the environment cannot run them. Report any command that was not run and why. Run `go mod tidy` only after dependency changes because it can create unrelated module-file churn. Run `sqlc generate` only when the relevant SQL or generation configuration changed, and review the generated diff.

## Change checklist

Before considering API work complete:

1. Read this file and inspect the nearest existing feature for local conventions.
2. Keep transport, use-case, and persistence responsibilities at their respective boundaries.
3. Add or update focused tests for behavior and failure paths.
4. Format changed Go files with `gofmt`.
5. Run targeted tests, then `go test ./...` and `go vet ./...` when practical.
6. Update `docs/openapi.yaml` whenever the externally visible HTTP API changed.
7. Regenerate sqlc output when SQL changed and verify generated files were not edited manually.
8. Check the diff for secrets, plaintext credentials, accidental generated changes, and unrelated formatting churn.
